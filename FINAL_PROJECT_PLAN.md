## 📋 执行摘要

### 项目名称
**OpenClaw NapCat Channel** - 基于 NapCatQQ 的完整功能 QQ 频道插件

### 核心价值
1. ✅ **完整消息支持** - 支持所有 QQ 消息类型的收发和解析
2. ✅ **历史消息存储** - 使用 SQLite 持久化存储所有消息
3. ⏳ **Block Streaming** - 边生成边发送（待实现）
4. ⏳ **输入状态显示** - 私聊原生 API + 群聊名片后缀（待实现）
5. ⏳ **智能交互** - 借鉴 MaiBot 理念，实现类人交互（Phase 2）

### 开发周期
**10 周**（Phase 1: 4 周 ✅ + Phase 2: 4 周 ⏳ + Phase 3: 2 周 ⏳）

## 七、现状复核与重构分析（2026-03-13）

> 说明：本章节用于补充对当前实现状态的复核结论，不改变第二章中的需求定义与优先级，只更新“当前实现是否真正闭环、工程边界是否清晰、后续应如何重构”。

### 7.1 复核结论摘要

1. **项目定位成立，但当前实现更像“已打通主流程的原型”而不是“边界清晰的 Channel 插件”**  
   `napcat-openclaw` 的核心价值依然成立：把 NapCat / OneBot v11 接入 OpenClaw，使 Agent 能通过 QQ 私聊、群聊、多模态消息与用户交互。

2. **需求层“已完成”与工程层“已收口”不是一回事**  
   Phase 1 中不少能力从“能跑通主链路”的角度可以视为完成，但从运行时契约、配置真源、模块边界、可测试性来看，仍存在明显技术债务。

3. **本仓库应收敛为 Channel 适配器，`napcat-tools` 应收敛为主动 API 工具层**  
   两个仓库应保持独立，通过接口契约对齐，而不是继续复制实现。

### 7.2 核心功能的重新定位

从现有源码看，当前项目的核心功能可归纳为 4 层：

- **连接适配**：通过 WebSocket 连接 NapCat，并发送/接收 OneBot 事件。
- **消息归一化**：将 QQ 消息段、通知事件、多媒体信息转换为 OpenClaw 可消费的上下文。
- **会话编排**：根据私聊/群聊、触发条件、上下文回捞、访问控制决定是否将消息交给 Agent。
- **辅助持久化**：通过 SQLite 保存消息和相关元数据，支撑历史查询与上下文补全。

这意味着本项目本质上不是“通用业务层”，而是 **OpenClaw 的 NapCat Channel 适配器**。后续重构应围绕“适配器职责清晰化”展开，而不是继续在插件仓库中扩展人格、策略、学习等业务能力。

### 7.3 基于外部文档的架构复核

#### OpenClaw Channel 开发指南给出的约束

- Channel 插件推荐按 `config`、`gateway`、`messaging`、`outbound`、`security`、`normalize` 等职责分层。
- `config` 与 `outbound` 是一等适配器，`gateway`、`status`、`messaging`、`security` 等为清晰可测的模块，而不应全部集中在单个入口文件里。
- 媒体能力应优先作为明确的出站能力建模，而不是散落在多个隐式调用路径中。

#### NapCat API 文档给出的约束

- `send_private_msg`、`send_group_msg` 是正式发送接口。
- `file`、`record`、`video` 等消息段是正式支持的消息类型。
- `delete_msg`、`get_msg`、`set_group_ban`、`set_group_kick`、`send_poke`、`ocr_image` 等能力均已有 API 支撑，后续扩展接口应优先“接通已有能力”而不是重新设计需求。

### 7.4 当前实现的关键问题

#### 7.4.1 生命周期事件契约失配（P0）

- `src/client.ts` 发射的是 `connected` / `disconnected` 事件。
- `src/channel.ts` 监听的是 `connect` / `disconnect` / `heartbeat`。
- 结果是 `probeAccount`、启动后的登录信息获取、活动记录、断开处理都存在潜在失效风险。

**结论**：该问题属于正确性缺陷，优先级高于新增功能开发。

#### 7.4.2 构建入口与配置入口分裂（P0）

- `package.json` 声明主入口是 `dist/index.js`。
- `tsconfig.json` 只编译 `src/**/*`，根目录 `index.ts` 不在构建范围内。
- 插件配置同时存在于 `index.ts`、`src/channel.ts`、`openclaw.plugin.json` 三处，已经形成多份真源。

**结论**：当前构建产物与宿主实际加载入口存在不确定性，配置也存在继续漂移的风险。

#### 7.4.3 `channel.ts` 已成为上帝模块（P1）

当前 `src/channel.ts` 同时承担：

- 账号启动/登出
- 连接事件监听
- 入站消息处理
- 触发判断
- 访问控制
- 群聊上下文回捞
- Agent reply 分发
- typing 状态控制
- 数据库存储

**结论**：该结构虽然便于早期快速开发，但后续继续在此文件上叠加能力，维护成本会迅速上升。

#### 7.4.4 配置已定义但未完全接线（P1）

`typing.enabled`、`typing.privateChat`、`typing.groupChat`、`typing.nicknameSuffix`、`typing.delayMs` 等配置已经设计完成，但主流程仍有明显硬编码行为，例如：

- 收到触发消息后无条件开始 typing；
- 群名片后缀与私聊刷新间隔仍依赖模块内部默认值；
- 路径和媒体处理存在隐式兜底逻辑。

**结论**：当前配置优化在“表单可见性”上完成度较高，但在“运行时消费完整度”上仍需补齐。

#### 7.4.5 消息解析存在双实现与语义漂移（P1）

- `src/message/parser.ts` 中有一套 `resolveMessageText()`。
- `src/utils/message-resolver.ts` 中又有一套 `resolveMessageText()`。
- 主流程当前使用的是 `utils` 版本，另一个实现仍保留并继续演化。

**结论**：这是典型的重复实现债务，后续支持更多消息类型时会越来越难维护。

#### 7.4.6 notice 事件被过度改写为“伪消息”（P1）

当前实现会将多种 notice 事件统一改写为 message 流处理，例如：

- 戳一戳
- 群文件上传
- 精华消息
- 群成员变动
- 群禁言
- 群名片修改

**结论**：这种做法短期统一了处理链路，但也模糊了“真实聊天消息”和“系统事件”的边界，不利于后续做结构化上下文和细粒度控制。

#### 7.4.7 文档、计划与实现状态存在漂移（P2）

复核后发现以下典型不一致：

- README 中 Node 版本要求与 `package.json` 的 `engines.node` 不一致；
- README 写有 `npm test`，但仓库中尚无对应测试脚本；
- 计划书当前写有“可按内容搜索消息”，但数据库实现中未见对应正式接口；
- 一些“已完成”条目仍更接近“链路已打通”，还未达到“结构清晰、可维护”的完成标准。

**结论**：后续需要将“需求完成状态”和“工程完成状态”分开描述。

### 7.5 与 `napcat-tools` 的关系重估

#### 7.5.1 `napcat-tools` 当前定位

`napcat-tools` 更适合作为 **NapCat 主动 API 的工具层/Skill 仓库**，而不是本仓库的共享实现层。它当前主要承担：

- 主动发送消息
- 发送文件/语音/视频
- 下载文件
- 查询历史消息

#### 7.5.2 当前耦合问题

两个仓库当前没有正式共享包，但存在事实上的“协议耦合 + 逻辑复制”：

- 都直接依赖 NapCat action 名和参数结构；
- 都处理历史消息查询；
- 都有消息解析语义，但并未共享同一实现；
- `napcat-tools` 文档中列出的部分命令并未真正实现。

#### 7.5.3 重构边界建议

后续应明确：

- `napcat-openclaw`：只负责 Channel 接入、入站/出站归一化、运行时生命周期、必要的 transport 特性。
- `napcat-tools`：只负责主动命令、文件路径处理、JSON 输出和少量工具层格式化。
- 两者通过 **命令契约、DTO 结构、返回格式** 对齐，不再共享实现。

### 7.6 不改变需求前提下的重构原则

本项目后续重构遵循以下原则：

1. **不改变第二章中的需求范围与优先级**  
   基础管理功能、智能交互、高级功能仍按既定规划推进。

2. **先修正确性，再做大重构**  
   生命周期事件、构建入口、配置真源问题优先级高于新增功能。

3. **先收敛适配器边界，再扩展业务能力**  
   在 `napcat-openclaw` 没有回到标准 Channel 插件结构前，不建议继续把人格、策略、学习等能力放进主链路。

4. **保留仓库独立性，不抽共享实现包**  
   当前阶段更适合通过契约文档和测试保持一致，而不是引入新的共享基础库。

### 7.7 重构优先级建议

#### 第一阶段：运行稳定性止血

- 修复 `client.ts` 与 `channel.ts` 的事件名不一致问题；
- 为主动断开增加显式状态，避免误重连；
- 在登出时关闭数据库并清理资源；
- 收敛日志输出，减少运行噪音。

#### 第二阶段：入口与配置收口

- 统一插件构建入口；
- 统一配置 schema 真源；
- 确保 Dashboard、运行时和文档看到的是同一套配置定义；
- 将 typing 等配置完整接入运行链路。

#### 第三阶段：适配层拆分

- 将 `channel.ts` 拆为 `gateway / messaging / outbound / security / normalize` 等模块；
- 合并重复消息解析逻辑；
- 重新定义 notice 与 message 的归一化边界。

#### 第四阶段：双仓库协同清理

- 收缩 `napcat-tools` 为稳定工具层；
- 修复其下载逻辑、命令实现与文档不一致问题；
- 为双仓库建立测试与契约文档。

### 7.8 对 Phase 2 / Phase 3 的影响说明

本次复核后，**需求本身不需要改动**，但建议调整后续开发顺序：

- 在进入 Phase 2 智能交互前，先完成至少第一、第二阶段的基础重构；
- 否则“发言决策、情感系统、动作系统、学习系统”等新能力会叠加在不稳定的底座之上；
- Phase 3 中的扩展接口应优先复用 NapCat 已有 API，而不是从零设计实现路径。
